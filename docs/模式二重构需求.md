# 模式二交互序列重构

## 目标
修复模式二（主题驱动）的交互顺序，让每一步该问的都问到位，不该自动跳过的绝不跳过。

## 新流程

```
主题输入 → 出文案(完整展示) → 用户确认文案 → 
素材需求(分图片/视频两块展示) → 问用户有素材吗？ → 
有→分析素材→问图片还是视频→生成
无→问图片还是视频→生成
```

## 修改点

### 1. mode_topic_driven() — 主流程重构 (main.py ~line121)

**原流程：**
```
步骤2: 出文案 → 只打印150字预览 → 步骤3: 素材需求 → 步骤4: 等用户给素材
→ 有空转到 _handle_no_materials（先出卡片再问图/视频）
```

**新流程：**
```
步骤2: 出文案 → 完整展示[标题][正文][标签]
步骤3: 用户确认文案
  - 输入 y → 继续
  - 输入 n → 让用户输入修改要求 → 重新生成文案
步骤4: 素材需求（分图片/视频两模块展示）
步骤5: 问用户是否有素材？
  - 是 → 用户输入路径 → 分析 → 问图/视频 → 生成
  - 否 → 问图/视频 → 生成
```

**具体改动：**

a) 步骤2打印完整文案，不是150字预览
b) 新增步骤3：用户确认（y/n + 可选修改要求）
c) 步骤4素材需求改为调用新方法，分图片/视频展示
d) 步骤5改为明确的"有素材吗？"提问，不再依赖空路径判断

### 2. 新增 generate_material_requirements() — 结构化素材需求 (在 copy_engine.py 或 main.py)

接收文案内容，输出分两块的素材清单：

```
📷 图片素材要求：
  【素材1】xxx | 用途
  【素材2】xxx | 用途

🎬 视频拍摄要求：
  【镜1】内容 | 时长 | 拍摄建议
  【镜2】内容 | 时长 | 拍摄建议
```

用 DeepSeek 生成，但输出格式由代码结构化，不是原始AI文本。

### 3. _handle_no_materials() — 先问再生成 (main.py ~line639)

**原逻辑：**
```
分割段落 → 问引擎 → 生成封面+卡片 → 问图片还是视频（卡片已经生成了）
```

**新逻辑：**
```
问「要图片还是视频？」 
  → 图片：生成封面+卡片（文案要点数决定卡片数）
  → 视频：生成封面+卡片 → 分镜 → 配音 → 合成MP4
先问再动手，不浪费资源。
```

### 4. 去掉硬编码的"工具猫"水印

_handle_no_materials 第711行 `f"工具猫 · {topic}"` → 改为只传 topic

### 5. 素材需求清单方法 — 新增专用方法

在 copy_engine.py 或新增方法中，调用 DeepSeek 生成结构化素材需求。
输出格式由代码确保分图片/视频两块，不依赖AI自觉。

## 不改的部分
- 自存参考库逻辑（save_output 已正常工作）
- _handle_images_only, _handle_multi_video, _handle_single_video 的内部逻辑
- providers/ 和 modules/ 层的代码
- 参考库读写

## 验证方式
1. 运行 mode_topic_driven('穿搭')
2. 确认文案完整展示
3. 确认 y/n 确认步骤生效
4. 确认素材需求分图片/视频两块
5. 选择无素材→问图/视频→正确生成
6. 选择有素材→分析→问图/视频→正确生成
